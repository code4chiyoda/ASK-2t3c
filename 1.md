# Asking Stakeholders for Knowledge 2t3c 提案プロセス

## 要約

Asking Stakeholders for Knowledge 2t3c は、地域コミュニティに情報を提供する文書です。ASK-2t3c の目的は、コミュニティの課題解決に貢献することです。

この文書は、ASK-2t3c の公開範囲、形式、プロセスについて説明しています。

## 範囲と目標

ASK-2t3c はコミュニティにとって関心のある話題に関して、簡潔かつ焦点を絞った文書化を行います。

具体的な内容について特に定めませんが、ウェルビーイングの向上を常に意識すべきです。

文書には以下が含まれることが期待されます。

- 課題やニーズの説明
- 解決手段の提案
- 技術的プロトコルやメカニズム
- 面白くてユーモラスなことがら

## 言語、文書構造、様式

提案はMarkdownでフォーマットされなければなりません。

言語について特に定めませんが、忘れないでください: これは情報提供書です。被提供者やメンテナが理解できなければ役に立ちません。

著者は包括的な言葉や例を使い、抑圧的な言葉遣いを避けるべきです。

## 提案タイトル、識別子

すべての文書には説明的なタイトルが必要です。

タイトルから、sha256ハッシュを算出し、冒頭4桁を識別子とします。標準のUnixツールで計算することができます。

```bash
echo -n 'Asking Stakeholders for Knowledge 2t3c 提案プロセス' | sha256sum | cut -c-4
0e99
```

ハッシュを使用することで、ファシリテーターが固有のIDを割り当てる負担を減らすことができます。

タイトルが固有のものである必要があります。

# 1. 動機

*背景事情**: 現在私は『千代田区福祉交通「風ぐるま」乗換案内』を開発中ですが、一人でぜんぶやるのは無理があります。

*それはなぜ？**: このソフトウェアについては、私自身が当事者です。しかしながら実際に運用する場合、それだけではだめです。他の当事者の知見が必要です。

*ではどうするのか？**: 情報提供と意見の募集を目的とした文書を作成しようと思いました（RFC のような）。

# 2. 手法

## 2.1 管理

 書は Git 管理し、任意のプロバイダでホストします。現在、[Github](https://github.com/code4chiyoda/ASK-2T3C) でホストしています。

### 2.2 提供と募集

以下のような手段があるかもしれません:

- メール
- 印刷
- HTML

なるべく古典的な手段を残したいです。**なぜ？**: 協働する人が電子機器に詳しいとは限りません。

## 3. ID

`ASK-2T3C-1` のように一意の ID を振ります。

### 3.1 ASK（接頭辞）

**A**sking **S**takeholders for **K**nowledge (当事者に知見を求める) という意味です。何か思いついたら、それが妥当か当事者の見解を確かめる必要があります。そのための文書です。

**なぜ汎用なのか？**: 過度に汎用的と思われるかもしれません。しかしながら、情報提供したいことや聞きたいことがあれば、どの用途に適用してもいいと思いました。

**それでは用途が不明ではないか？**: 次の記号で用途を特定します。

**なぜ RFC ではないのか？**: RFC は非常に良いネーミングです。しかしながら、既に IETF の情報提供文書を示す固有名詞になっています。勝手に使うとまぎらわしいです。

### 3.2 2T3C （用途記号）

`2T3C` は、短く一意に千代田区を示すための記号で、あまり意味はありません。千代田区の地方公共団体コード `131016` を Base36（`0-9` と `A-Z` をぜんぶ使う数字の記法: 36 進数）で表した値です。

### 3.3 採番

`1` のように通し番号をつけます。 `001` のようにゼロ埋めはしません。
