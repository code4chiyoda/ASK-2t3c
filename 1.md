# 知見求む（ASK 文書）について

## 1. 動機

**背景事情**: 現在私は『千代田区福祉交通「風ぐるま」乗換案内』を開発中ですが、一人でぜんぶやるのは無理があります。

**それはなぜ？**: このソフトウェアについては、私自身が当事者です。しかしながら実際に運用する場合、それだけではだめです。他の当事者の知見が必要です。

**ではどうするのか？**: 情報提供と意見の募集を目的とした文書を作成しようと思いました（RFC のような）。

## 2. 手法

### 2.1 管理

文書は Git 管理し、任意のプロバイダでホストします。現在、[Github](https://github.com/code4chiyoda/ASK-2T3C) でホストしています。

以下のような選択肢を見つけました:

- [GitLab](https://gitlab.com/users/sign_in)
- [tangled](https://tangled.org/)
- [NostrGit](https://github.com/NostrGit/NostrGit?tab=readme-ov-file)

### 2.2 提供と募集

以下のような手段があるかもしれません:

- メール
- 印刷
- HTML

なるべく古典的な手段を残したいです。**なぜ？**: 協働する人が電子機器に詳しいとは限りません。

### 2.3 ライセンス

とりあえず GPL ライセンスにしてみました。私はなにかをオープンソースにしようと思ったら、とりあえずコピーレフトにします。反対意見があれば変えるかもしれません。

## 3. ID

`ASK-2T3C-1` のように一意の ID を振ります。

### 3.1 ASK（接頭辞）

**A**sking **S**takeholders for **K**nowledge (当事者に知見を求める) という意味です。何か思いついたら、それが妥当か当事者の見解を確かめる必要があります。そのための文書です。

**なぜ汎用なのか？**: 過度に汎用的と思われるかもしれません。しかしながら、情報提供したいことや聞きたいことがあれば、どの用途に適用してもいいと思いました。

**それでは用途が不明ではないか？**: 次の記号で用途を特定します。

**なぜ RFC ではないのか？**: RFC は非常に良いネーミングです。しかしながら、既に IETF の情報提供文書を示す固有名詞になっています。勝手に使うとまぎらわしいです。

### 3.2 2T3C （用途記号）

`2T3C` は、短く一意に千代田区を示すための記号で、あまり意味はありません。千代田区の地方公共団体コード `131016` を Base36（`0-9` と `A-Z` をぜんぶ使う数字の記法: 36 進数）で表した値です。

### 3.3 採番

`1` のように通し番号をつけます。 `001` のようにゼロ埋めはしません。

## 4. セキュリティについての考慮事項 

今は私が一人で書いているので、特にありません。スケールした場合、何か考え始める必要があります。
